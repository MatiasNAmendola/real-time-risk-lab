## Blow-by-Blow

### 1 — Problem Space Analysis
Dissected the production-grade fraud detection use case requirements. Key constraints: 150 TPS sustained, p99 < 300ms end-to-end (see [[Latency-Budget]]), hybrid sync+async architecture, Lambda→EKS migration path (see [[0007-k3d-orbstack-switch]]), Java + Spring Boot, AWS (EKS, SQS, SNS), [[Clean-Architecture]] / [[Hexagonal-Architecture]], online ML (see [[ML-Online-Fallback]]), team mentoring context. These constraints shaped every subsequent PoC and doc.

### 2 — Architecture Alignment
Alignment on what production-grade fraud detection systems value: systems thinking over code heroics, trade-off articulation, mentoring posture. Crystallized the framing: "I build systems that degrade gracefully, not ones that fail silently." Anchors [[Architectural-Anchors]].

### 3 — PoC Bare-javac (java-risk-engine)
First PoC: `poc/java-risk-engine/` using Java 25 bare `javac` (no Maven/Gradle, no Spring). Implemented [[Clean-Architecture]] with domain entities (`RiskScore`, `Transaction`), ports (repository + event publisher), use case (`EvaluateRiskUseCase`), adapters (HTTP controller, in-mem repository), [[Circuit-Breaker]] manual implementation, ML fallback (see [[ML-Online-Fallback]]), [[Idempotency]] guard, [[Outbox-Pattern]] stub. Proved architecture compiles and domain logic is framework-free.

### 6 — Enterprise Go Layout Research
Studied canonical enterprise Go monorepo conventions for `internal/{domain,application,infrastructure}` layout. Key learnings: layout matching [[Clean-Architecture]] terminology, custom resilience patterns (circuit breaker, retry, bulkhead implemented in-house), Kafka-compatible event bus, OTel observability stack, container-based integration testing, manual DI without framework. Identified [[Outbox-Pattern]] as a frequent gap in such layouts — added explicitly. See [[0008-outbox-pattern-explicit]].

### 7 — Refactor to Enterprise Go Layout
Refactored `poc/java-risk-engine/` to mirror enterprise Go layout: `domain/{entity,repository,usecase,service,rule}`, `application/usecase/risk/`, `application/mapper/`, `cmd/`, `config/`, `infrastructure/{controller,consumer,repository,resilience,time}`. All tests pass. Benchmark: p50=50µs, p95=127ms, p99=153ms, 1528 req/s. See [[0002-enterprise-go-layout-in-java]] and [[java-risk-engine]].

### 8 — Vert.x Distributed PoC
Built `poc/java-vertx-distributed/` (later renamed `vertx-risk-platform/`): 4 Maven modules — `shared`, `controller-app`, `usecase-app`, `repository-app`. Vert.x 5 with Hazelcast TCP [[Layer-as-Pod]] cluster manager. 4 Docker networks (`ingress/eventbus/data/telemetry`). UIDs distinct per container. OTel Java agent 2.x. See [[0003-vertx-for-distributed-poc]] and [[java-vertx-distributed]].
